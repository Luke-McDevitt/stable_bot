Active Stabilization (Ball-Hold) (scaffold for v10)

Keeps the ball at its current plate-frame position despite base perturbations (someone bumps the table, bot on a wobbly cart, slow tilt). Combines the existing plate-level loop with /base/imu/data feedforward and vision feedback. Workspace is limited to ~30 mm of top-plate translation; falls back to plate-frame holding past that.

Implementation deferred to v10. Math source of truth: stewart_vision/_ball_physics.py (feedforward_tilt_for_base_accel, saturation_fraction; both unit-tested). Design: spec §11.7. Reserved mode token: BALL_HOLD.

Legacy: open-loop rolling ball (advanced — superseded by Demo 1)

Generates a precomputed tilt-rotation routine. Kept for fallback / regression testing only — Demo 1 (Orbit, closed-loop) is the primary orbital demo now.

  • Ramp-up: spin-up phase. The orbit radius R grows 0 → R_final via a smoothstep. During this phase the tilt includes a tangential component (phase lead) that physically pushes the ball along the orbit, building up angular momentum. Shorter = more aggressive kick; longer = gentler.
  • Hold cycles: full orbit rotations at the steady-state tilt (R̈=Ṙ=0, just centripetal).
  • Ramp-down: mirror of ramp-up. R shrinks R_final → 0 with a braking phase lead that drains the ball's tangential velocity back to rest.
  • Center hold: flat platform (tilt = 0°) so the ball can come to rest on top before the routine ends.
  • Min tilt: safety floor during the active phases (not applied during center hold). Usually leave at 0 with the new feed-forward; only useful if you see the ball sitting still during early ramp-up because of rolling friction.

Set ramp-up or ramp-down to 0 to disable it (instant jump to/from the hold tilt — useful for breaking stiction but motors will slew hard).

Tip: at R=180 mm the min safe period under the 15°/s cap is ~3 s. First try without a ball (do not put a ball on it until you like what you see).

CAN recovery

If that fails → manual steps
  1. From Windows admin PowerShell:
    usbipd list usbipd detach --busid <X-Y> usbipd attach --wsl --busid <X-Y>
  2. If the adapter's LED stayed dark during detach/attach, physically unplug the USB cable and re-plug it, then re-run usbipd attach.
  3. In WSL: sudo ip link set can0 up type can bitrate 1000000
  4. Click "Soft-reset CAN bus" again.

To enable the soft-reset button (passwordless sudo for ip link):

sudo visudo -f /etc/sudoers.d/stewart-can

Homing

Two-step workflow. Step 1 stall-homes the bottom endstops automatically. Step 2 is a manual top-of-stroke capture — you physically move each leg to its top position, then press [ENTER] in the prompt to record it.

Pre-homing safety

Snapshot the current encoder readings as the rest reference so the regular Arm button doesn't refuse (when no leg_limits.yaml exists yet) and doesn't drive into the endstops. Two-step flow:

  1. Manually position each leg to a safe spot (mid-stroke).
  2. Click 📍 Capture rest below — just records positions, doesn't arm.
  3. Then press the regular Arm button in the left column.

Manual endstop capture (recommended)

No motor movement — physically position the legs and click. Replaces the automatic stall-homing flow below; no risk of slamming into endstops.

  1. Position all 6 legs at the BOTTOM (resting on foam blocks, just above the actual endstop).
  2. Click 📍 Capture BOTTOM.
  3. Position all 6 legs at the TOP (fully extended upward).
  4. Click 📍 Capture TOP.
  5. Adjust the rest-offset slider if needed (default 0.20 turns ≈ 1 mm above bottom).
  6. Click 💾 Save limits — writes leg_limits.yaml, no homing script needed.
